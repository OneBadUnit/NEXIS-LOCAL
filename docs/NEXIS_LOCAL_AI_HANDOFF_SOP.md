## 1. What NEXIS-LOCAL Is

NEXIS-LOCAL is a **single-user, local-machine AI workspace** for information collection, transformation, and creator output generation.

It is designed to run entirely on one user's machine. All compute, storage, and AI inference happen locally. There is no cloud dependency required for the core workflow to function.

**Core workflow:**

```
Collect → Select → Create → Refine → Reuse
```

**What this means in practice:**

- The user collects raw information from documents, URLs, images, audio, and video.
- The user selects which collected material feeds into a generation run.
- The system produces structured outputs: summaries, timelines, outlines, creator scripts, hooks, and keywords.
- The user can refine those outputs without regenerating from scratch.
- Outputs can be reused across projects.

**Runtime stack — all local:**

| Layer | Technology | Notes |
|---|---|---|
| Frontend | React (port 3000) | Runs locally in browser or Electron |
| Backend | FastAPI (Python) | Runs locally, no public exposure required |
| Database | SQLite (`nexis.db`) | Local file, auto-created on first run |
| AI inference | Ollama (`localhost:11434`) | Local model server |
| Bridge | NEXIS Companion (`localhost:8765`) | Local process managing Ollama safely |
| OCR | Tesseract (local binary) | Bundled path override via env var |

## 2. What NEXIS-LOCAL Is Not

- **Not a SaaS product.** There is no hosted backend users connect to.
- **Not a multi-user platform.** No teams, no shared workspaces, no collaborative editing.
- **Not a cloud AI service.** No managed inference endpoints. The user runs their own models.
- **Not subscription-gated in practice.** Tier/plan logic exists in the codebase but all local-mode limits are set to effectively unlimited (`99999`). Billing is not enforced locally.
- **Not browser-to-Ollama direct.** The frontend never talks to `localhost:11434` directly. All AI communication goes through the NEXIS Companion bridge.
- **Not dependent on external APIs.** Provider API keys (OpenAI, etc.) are a secondary fallback option, not a requirement.

## 4. Core Workflows

### 4.1 Collect

The user ingests raw source material into a project. Supported source types:

- PDF documents
- DOCX files
- Plain text
- URLs (web extraction)
- Images (OCR + optional vision model description)
- Audio/video (transcription pipeline — in development)

For article and news URLs, NEXIS isolates the article body before storing source content — webpage boilerplate (navigation, sidebars, ads, related articles) is filtered at ingestion time, before the source reaches any generation pipeline.

Each source is stored in the local SQLite database attached to a project.

### 4.2 Select

The user chooses which collected sources feed into a package generation run. Multi-source selection is supported. Source boundaries are preserved in the output.

### 4.3 Create (Package Generation)

Two package types:

**Summary Package** — structured analysis output:
- Summaries
- Timelines
- Key Points
- Outlines
- Thematic synthesis across multiple sources

**Creator Package** — creator-ready output:
- Hook Scripts
- Video Script Frameworks
- Dialogue Scripts
- Title Suggestions
- Keywords
- Engaging rewrites

Packages are generated by the local LLM (default: `qwen2.5:7b`) via the NEXIS Companion bridge and FastAPI backend.

### 4.4 Refine

Previously generated outputs can be refined without regenerating the full package. Refinement modes include tone changes, length adjustments, format shifts, and SEO optimization. Each refinement is stored as a new output record.

### 4.5 Reuse

Projects and their outputs persist in SQLite. The user can return to prior work without re-ingesting sources.

## 5. Current Architecture Assumptions

These are the verified architectural facts as of this document's creation date. Do not infer or override these without direct file evidence.

### Frontend

- React application, port `3000`
- Communicates with the FastAPI backend and the NEXIS Companion bridge
- All Ollama calls route through `bridge.js` → NEXIS Companion → Ollama
- Direct browser-to-Ollama (`localhost:11434`) communication is explicitly prohibited by design
- UI state uses React context (`ArcNContext`) and `localStorage` for lightweight persistence
- `localStorage` access is wrapped with try/catch protection due to restricted-context edge cases

### Backend

- FastAPI, Python
- Default database: SQLite (`nexis.db`, local file, auto-created)
- SQLAlchemy ORM — SQLite connection uses `check_same_thread: False`
- CORS is configured to allow `localhost:3000` for local development
- Feature flags control optional capabilities: `WHISPER_ENABLED`, `OCR_ENABLED`, `VISION_ENABLED`, `YOUTUBE_INGESTION_ENABLED`, `NEXIS_HOSTED_MODE`
- All feature flags default to `False` — must be explicitly enabled via env vars

### AI / Model Layer

- Ollama runs locally, default port `11434`
- Default LLM: `qwen2.5:7b`
- Default vision model: `llava:13b`
- The NEXIS Companion (Go binary) manages Ollama lifecycle: detection, startup, model listing, generation
- The companion runs on `localhost:8765`
- If the companion is not running, AI features are unavailable — this is expected and surfaced to the user through diagnostics UI

### Storage

- Primary storage: SQLite (`nexis.db`) — local file
- No object storage (no S3, no Supabase Storage) required for core workflow
- `localStorage` used for UI state and onboarding persistence only

### Authentication

> **Note:** Authentication uses Supabase, which is a cloud service.  
> This is the one external cloud dependency present in the current architecture.  
> See Section 8 for context on this design decision.

- Auth is handled via `supabase.js` and `auth.js`
- Only email + password flows are supported (magic links and OTP are intentionally absent)
- Password reset flow uses Supabase email relay
- Profile data is stored in Supabase (cloud)

## 6. Local-First Principles

These principles govern all architectural and implementation decisions for NEXIS-LOCAL.

1. **The core workflow must not require an internet connection.** Collect, Create, Refine, and Reuse must function fully offline once the local stack is running.

2. **The user owns their compute.** AI inference is performed by the user's local hardware. No inference costs are billed to any service.

3. **The user owns their data.** Project data, source content, and generated outputs are stored locally in SQLite. No project data is sent to external services.

4. **No silent network calls.** The frontend must not contact any external host except through deliberate, user-initiated actions.

5. **Graceful degradation is required.** If a local service is unavailable (Ollama down, companion not running), the system must surface a clear diagnostic and remain usable for non-AI functions. It must not crash or produce silent failures.

6. **Minimal dependencies.** Add new dependencies only when clearly necessary. Prefer existing stdlib or established packages over novel integrations.

7. **Simple is correct.** MVP-focused, practical architecture. No enterprise abstractions, no premature scalability patterns.

## 7. Legacy Concepts Carried Forward From AEGIS

NEXIS-LOCAL evolved from prior ARC NEXUS work. The following architectural concepts were carried forward and remain present in the codebase.

### 7.1 Tier / Plan System

The `tiers.py` file defines plan tiers (`free`, `base`/Core, `middle`/Pro, and higher). This structure was designed for a hosted, multi-user product with enforced billing limits.

**In NEXIS-LOCAL:** All limits are set to `99999` (effectively unlimited). The tier system exists in the code but is not meaningfully enforced for a single local user. Do not add tier enforcement logic targeting local users unless explicitly instructed.

### 7.2 Usage Tracking

Usage counting logic exists in `core/usage.py` and the usage API route. This was designed for quota enforcement in a hosted context.

**In NEXIS-LOCAL:** Usage tracking may run but limits are not binding. Do not treat usage-tracking code as a behavioral constraint on the local user.

### 7.3 Supabase Authentication

Supabase was integrated as a managed auth service. It remains the current auth layer.

**In NEXIS-LOCAL:** This introduces the only required cloud dependency. The user must have a Supabase account and internet access to authenticate. This is a known design tension and is accepted for now.

### 7.4 CORS Configuration

`main.py` includes Vercel preview URLs in the `ALLOWED_ORIGINS` list. These were valid origins for the hosted frontend.

**In NEXIS-LOCAL:** Only `localhost:3000` and `127.0.0.1:3000` are the active origins. The Vercel entries are harmless remnants and should not be removed without explicit instruction.

### 7.5 The Bridge / Companion Architecture

The NEXIS Companion binary (Go) was designed to provide a safe local proxy between the frontend and Ollama. This concept was created as part of AEGIS-era local AI work and is carried forward as a core component.

**In NEXIS-LOCAL:** The bridge is a first-class component. It is not legacy — it is the correct and intended AI communication layer.

## 8. Concepts That No Longer Apply

The following concepts existed in earlier or hosted versions and **do not apply to NEXIS-LOCAL**.

### 8.1 Multi-Tenancy

AEGIS was designed with multiple users, organizations, or teams in mind. NEXIS-LOCAL is single-user. Do not reason about user isolation, tenant separation, or per-user data segregation. There is only one user.

### 8.2 Hosted Backend

There is no Render deployment, no cloud FastAPI server, and no production backend URL for NEXIS-LOCAL. The FastAPI backend runs locally only.

### 8.3 Managed Database

PostgreSQL with a cloud host is not the target database for NEXIS-LOCAL. SQLite is the correct default. Do not suggest PostgreSQL migrations for NEXIS-LOCAL workflows.

### 8.4 Token Billing / AI Cost Management

NEXIS-LOCAL does not bill the user for AI inference. There are no token limits, no rate limiting tied to billing, and no cost-per-call tracking. The user runs their own model.

### 8.5 Subscription Enforcement

Plan tier limits are not enforced for local users. Do not add code that gates features behind plan checks unless explicitly requested.

### 8.6 CDN / Object Storage

File uploads are handled locally. There is no S3 bucket, no Supabase Storage bucket, and no CDN. Uploaded files are processed in-process by the FastAPI backend.

### 8.7 Admin Dashboards / Analytics

No admin UI, no usage dashboards, and no telemetry are part of NEXIS-LOCAL.

## 9. Future Hosted NEXIS (Not Implemented)

> **FUTURE POSSIBILITY ONLY — NOT CURRENT ARCHITECTURE**

The following concepts may apply to a future hosted version of NEXIS. They are documented here so that AI assistants do not accidentally introduce hosted-mode assumptions into NEXIS-LOCAL code.

**Future hosted concepts (not implemented):**

- Multi-user accounts with real tier enforcement
- PostgreSQL as the primary database
- Cloud-hosted FastAPI backend (e.g., Render, Railway, Fly.io)
- Managed AI inference endpoints (OpenAI, Anthropic, or hosted Ollama)
- Supabase Storage for uploaded files
- Token/cost billing and quota enforcement
- Analytics and usage telemetry
- Team and organization workspaces
- Web-only access without a local companion requirement

**What the `NEXIS_HOSTED_MODE` flag means:**  
`config.py` contains `NEXIS_HOSTED_MODE: bool = False`. This flag was added to anticipate a future hosted deployment. It is `False` by default and must remain `False` in all local builds. Do not enable it or write code that assumes it is `True` unless working explicitly on the future hosted product.

## 10. Instructions For Future AI Assistants

Read this section carefully before beginning any work session on NEXIS-LOCAL.

### 10.1 Establish Context First

Before analyzing or editing any code:

1. Read this document.
2. Read `docs/ai/NEXIS_CONTEXT.md`.
3. Read `docs/ai/KNOWN_ISSUES.md`.
4. Read `ARC_CONTEXT.md`.
5. Check `NEXIS_TREE.md` for the current repository structure.

Do not infer architecture from filenames alone. Verify with direct file reads.

### 10.2 Scope Rules

- **Default scope is frontend unless the user explicitly expands it.**
- Do not modify backend files unless the task clearly requires it.
- Do not modify the database schema unless explicitly instructed.
- Do not modify `tiers.py`, `usage.py`, or auth logic unless the task specifically targets those systems.
- Treat Supabase auth as a stable external boundary — do not refactor it without explicit instruction.

### 10.3 Local-First Assumption

Every implementation decision must default to local-first behavior.

- If a feature could be done locally or with a cloud API, prefer local.
- If you are about to add a network call to an external service, stop and ask first.
- Do not add new cloud dependencies without explicit user approval.

### 10.4 MVP Discipline

This project is in MVP stabilization. Apply minimal-fix discipline to all changes.

- Fix the specific thing that is broken.
- Do not refactor surrounding code while fixing a bug.
- Do not add features that were not requested.
- Do not introduce new abstractions for one-time operations.
- Do not add enterprise patterns, global state managers, or premature optimizations.

### 10.5 Do Not Confuse Hosted and Local

Several files contain CORS origins for Vercel, references to Render, PostgreSQL configuration, and hosted-mode flags. These are **not the active configuration for NEXIS-LOCAL**. Do not use them as evidence that NEXIS-LOCAL is a hosted product. Do not modify them unless explicitly asked.

### 10.6 The Bridge Is Mandatory

The NEXIS Companion bridge on `localhost:8765` is a required local component, not an optional integration. When working on AI generation features, assume the bridge is the communication layer. Never suggest direct browser-to-Ollama communication as a fix or simplification.

### 10.7 Auth Is a Cloud Dependency — Accept It

Supabase auth is used. This means the user needs internet access and a Supabase account to log in. This is the current accepted design. Do not remove Supabase auth without explicit instruction. Do not add a second auth system in parallel.

### 10.8 SQLite Is the Target Database

All local storage is SQLite. Do not suggest PostgreSQL, Redis, or any other database for NEXIS-LOCAL unless the user explicitly requests it. Do not add any migration tooling that assumes PostgreSQL.

### 10.9 Preserve Existing Behavior

Unless the task explicitly says to change behavior:

- Preserve all existing imports and exports.
- Preserve all existing comments.
- Preserve all formatting and structure.
- Do not change public API shapes or response contracts.
- Return full files when rewriting — no placeholders, no omission markers.

### 10.10 State Uncertainty Explicitly

If you cannot verify a file's contents, say so. Do not infer what a file probably contains. Do not reconstruct unseen files. If a referenced file is not available, state: "File not provided. Cannot analyze."
